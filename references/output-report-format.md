# 分析报告格式规范（生成报告时读）

仅在用户明确要求生成报告时使用。把 Phase 2–4 的分析与探讨结论落成**一个** Markdown 文件，默认 `<repo>/paper-code-analysis.md`（用户可指定路径）。

## 质量标准
- **每条断言可溯源**：映射带 `[paper §X]`/`[code: file:line]`/`[config: file:key]`/`[guess]`（见 `paper-code-mapping.md`）。
- **每条映射带三态结论**：`[matched]`/`[gap]`/`[fundamental disagreement]`。
- **概览报告 `[guess]` 占比**作为整体可信度信号。
- 不臆断：探讨中未能确认的，写"待核实"，不要为了完整而编。
- 改进方向必须**可操作 + 排序**，每条挂到具体依据(哪条 gap/风险)。

## 章节模板

```markdown
# <论文标题> — 论文代码分析报告

## 0. 概览
- 论文：<标题 / arXiv / DOI>
- 代码：<repo URL 或路径> @ <commit/tag/分支>
- 生成时间：<date>
- 一句话总结：<这个工作做了什么、关键贡献>
- 映射可信度：<[guess] 占比 / 主要不确定点>

## 1. 论文核心创新
1. **<创新点>**：解决的问题 / 相对前人的 delta / 为什么 work
2. ...

## 2. 算法与公式拆解
### 关键公式
- **eq.N**：<符号定义>，<含义>，<在方法中的作用>
### 算法流程
<伪代码或步骤化，标输入/输出与关键超参>

## 3. 代码结构与静态分析
### 模块与职责
- `<module>`：<一句话职责>
### 关键配置与超参（带出处）
- optimizer / batch / schedule / augmentation / loss / model / tricks，逐项 + 出处
### 入口表
| 用途 | 启动命令 / 文件:行 | 读取的 config | 关键依赖 |
### 依赖图
<数据→模型→loss→optimizer→eval 主干 + 模块间依赖>
### 日志摘要与数值核对
<已有日志的关键指标 vs 论文数字，带 matched/gap/disagreement；无则写"无">

## 4. 论文 ↔ 代码 映射
### 创新点/算法总映射
| 论文元素 | 章节 | 代码模块 | 文件:行 | 一致性 |
### 公式逐项核对
| 论文公式 | 章节 | 代码位置(file:line) | 关键变量对应 | 一致性 |
<对 reduction/边界/eps/归一化/梯度流的核对结论>

## 5. 核心模块即插即用抽取
对每个核心模块：
- **最小依赖**：文件 / 第三方库 / 项目内工具函数
- **接口契约**：输入 shape·dtype·取值 / 输出含义 / 副作用
- **解耦点 + 最小可用片段**：复制哪几段、改哪几处
- **改造钩子**：想改/替换它，动哪个函数 / 哪个 config key

## 6. 潜在工程风险（高 → 低）
| 等级 | 位置(file:line) | 现象 | 影响 | 建议 |

## 7. 改进方向（排序）
1. **<建议>**：依据(哪条 gap / 风险 / 过强假设) + 具体怎么改 + 预期收益
2. ...
```

## 规模适配
- 小项目：全部写进一个 `.md`。
- 大项目（先问用户）：§4 映射表拆 `paper-code-analysis/mapping.md`，单模块详解拆 `paper-code-analysis/modules/<name>.md`，主报告保留摘要 + 链接。
- 只深读了部分模块时，在 §3 注明"其余模块仅做结构概述"，不要假装全覆盖。
