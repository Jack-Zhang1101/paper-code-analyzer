# 论文 ↔ 代码 映射方法论

paper-code-analyzer 区别于"纯代码理解"的核心：把论文每个创新/公式/算法，精确、可溯源地对到代码实现位置，并验证两者是否真的一致。

## 1. 溯源标签（内部 evidence ledger 必带）
- `[paper §X]` / `[paper eq.N]` / `[paper Table k]`：论文里显式存在
- `[code: path:lineno]`：代码里找到的实现
- `[config: path:key]`：配置里的值
- `[venue convention]`：按领域/会议惯例推断
- `[framework default]`：显式用框架默认值
- `[guess: 写明假设]`：最佳猜测，必须写推理依据

`[guess]` 越少越可信。某模块 `[guess]` 占比过高 → 论文欠规范或代码与论文脱节，在"改进方向/风险"点出。

注意：这些溯源标签是内部 evidence ledger 的硬要求。最终报告若使用 `obsidian-reading-note`，不要把 `[code: path:lineno]`、`[config: path:key]`、commit、本地路径直接写进正文；把它们转写为读者友好的"实现逻辑/一致性判断/风险"。

## 2. 三态一致性结论（每条映射给一个）
- `[matched]`：论文描述与代码一致
- `[gap, hypothesis: ...]`：有可测量差异/缺失，附原因假设（如"论文说 cosine，代码是 step decay，疑后期改动未回写"）
- `[fundamental disagreement, see ...]`：论文主张与代码实质矛盾，需研究者判断，不是"多跑几次"能解决

## 3. gap 分析：一句话背后藏的超参
论文"trained 100 epochs with AdamW"至少藏 8 个超参。对每个方法描述主动追问"真要跑起来还缺什么"，逐项去 code/config 找，对齐 static-analysis 第 2 节六块。论文有代码找不到 → `[gap]`；代码有论文没提 → "实现细节，论文未述"（复现注意点）。

## 4. 公式-代码逐项验证清单 ⚠️（最易错、最有价值）
对每个核心公式找到实现它的代码，逐项核对：
- **求和/连乘边界** ↔ 循环/张量维度边界（off-by-one、是否含 padding）
- **reduction**：论文 mean 还是 sum？代码 `loss.mean()`/`sum()`/`reduction='none'` 后怎么聚合
- **归一化因子**：分母 N / N-1 / batch×dim；温度/缩放 √d 位置对不对
- **数值稳定**：`eps` 加在该加处没（除法/log/sqrt/softmax）
- **梯度流**：该 `detach()`/`no_grad()`/`stop_gradient` 的项（teacher/target/EMA）切断没
- **广播/维度**：einsum/广播与公式下标含义一致？转置、维度顺序对不对
- **系数对应**：论文每个 λ/β/α/τ 对应代码哪个变量、取值多少、来源标签

内部可先产出"公式映射表"：
```
| 论文公式 | 章节 | 代码位置(file:line) | 关键变量对应 | 一致性 |
|---------|------|--------------------|-------------|--------|
| eq.3 对比损失 | §3.2 | model/loss.py:88 | τ=cfg.temp=0.07 [config] | [matched] |
```

## 5. 创新点 ↔ 模块映射
把每条核心创新对到实现它的模块/函数：
```
| 论文创新点 | 章节 | 承载模块 | 关键实现(file:line) | 一致性 | 备注 |
|-----------|------|---------|--------------------|--------|------|
```
这是报告中"总体映射"的来源，也是"即插即用抽取"优先级依据（承载创新的模块优先抽）。默认 `obsidian-reading-note` 不直接输出宽表，应改成纵向 bullet；`engineering-trace` 才保留表格和 file:line。

## 6. round-trip 往返校验（防过度自信）
映射初稿后反向核对：每条 `[paper §X]` 回论文确认确有其说；每条 `[code:…]` 回代码确认确有其实现。对不上的降级为 `[guess]` 或改标 `[gap]`/`[disagreement]`。把"转录错误"和"过度自信归属"在进报告前拦下（ABC 自检的"映射题"会再查一遍）。

## 7. 即插即用抽取（基于映射）
对承载创新的核心模块整理成可复用形式：
- **最小依赖清单**：需要哪些文件、哪些第三方库、依赖项目内哪些工具函数
- **接口契约**：输入张量/参数的 shape、dtype、取值范围；输出含义；副作用
- **解耦点 + 最小可用片段**：复制哪几段、改哪几处路径/依赖就能在别处跑；需要拆掉的耦合（全局 config、注册表、相对 import）
- **改造钩子**：想替换/改进该组件（新方法设计），改哪个函数、哪个 config key

输出到报告的"核心机制/可复用思路"部分。默认 `obsidian-reading-note` 写机制和接口，不写具体文件路径；`hybrid` 可在附录列出精简证据；`engineering-trace` 可写完整依赖路径。
